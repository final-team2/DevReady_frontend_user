# CLAUDE.md — DevReady 협업·안전 작업 규칙 (AI·백엔드·프론트 3레포 공통)

> 이 저장소에서 작업하는 **모든 사람과 모든 AI(Claude 포함)는 작업 시작 전에 이 문서를 끝까지 읽는다.**
> **목적**: 누가 작업하든 — 무료 요금제·Sonnet·사고(thinking) 끄기·저사양 설정이라도 — **충돌·데이터 손실·범위 이탈 없이** 안전하게 기여하기 위함.
>
> - **Claude Code(터미널/IDE)** 는 이 파일을 매 세션 자동으로 읽는다.
> - **웹(claude.ai)** 은 자동 로드가 안 된다 → 8번 템플릿을 복붙해서 지시한다.
> - 이 문서는 **3개 레포 공통**이다. 작업 전 **7번 "레포 판별"** 로 자기 레포 규칙을 먼저 확인한다.

---

## 0. 이것만은 반드시 (TOP 3)

1. **지시받은 파일만 수정한다.** 그 외 파일은 읽기만. 수정·삭제·이동·이름변경·자동포맷 금지.
2. **`main`에 직접 푸시하지 않는다.** 반드시 본인 브랜치 → Pull Request. `--force` 금지.
3. **막히거나 애매하면 멈추고 사람에게 묻는다.** 추측으로 진행하지 않는다.

---

## 1. 절대 금지 — HARD STOP (예외 없음)

- ❌ 지시 범위 밖 파일 변경 (리팩터링·정리·"개선"·주석 정리·코드 스타일 변경 포함)
- ❌ `git push --force` / `git reset --hard`(공유 브랜치) / `git rebase`(공유 브랜치) / `rm -rf`
- ❌ `main`·`develop` 브랜치에 직접 커밋·푸시
- ❌ 비밀값 커밋: `.env`, API 키, 토큰, 비밀번호, 모델 가중치(`.bin`/`.safetensors`), 대용량 데이터셋
- ❌ AI 고정 스택 변경 (**7번 — AI 레포** 목록) — 버전·`revision` 절대 변경 금지
- ❌ 다른 팀원 담당 모듈 수정 (5번 소유권 맵)
- ❌ 레포 전체에 포맷터/린터 실행 (Prettier·Black 등으로 남의 파일까지 변경)
- ❌ 기존 클래스·파일 이름을 담당자 승인 없이 변경

위 중 하나라도 필요해 보이면 → **실행하지 말고 담당자에게 먼저 확인.**

---

## 2. 작업 5단계 (충돌 방지 표준 흐름)

1. **받기** — `git checkout main && git pull origin main` (항상 최신부터 시작)
2. **브랜치** — `git checkout -b feat/<이름>-<작업요약>`  (예: `feat/jsc-star-scoring`)
3. **작업** — 본인 담당 파일만. 작게, 자주 커밋.
4. **올리기** — `git add <변경한 파일만>` → `git commit` (메시지는 3번 형식) → `git push origin feat/...`
   - `git add -A` / `git add .` 는 의도치 않은 파일까지 올라간다. **변경한 파일을 명시**해서 add.
5. **PR** — Pull Request 생성. **한 PR = 한 기능/한 관심사.** 담당 영역이 겹치면 리뷰 후 머지.

> 같은 파일을 두 사람이 동시에 고치면 충돌이 난다. **소유권 맵(5번)을 지키면 충돌이 원천 차단된다.**
> 충돌이 났다면 → `rebase`로 풀지 말고 **멈추고 담당자와 함께 처리.**

## 3. 커밋 메시지 형식 (구체적·상세하게)

제목만 달랑 쓰지 않는다. **본문에 "왜·무엇을·영향 범위"가 반드시 들어간다.**

```
<타입>: <무엇을 했는지 한 줄 요약>      ← 제목 (한국어, 50자 내외)

- 왜 이 변경이 필요했는지 (배경/문제)
- 무엇을 어떻게 바꿨는지 (핵심 변경점)
- 관련 기능 ID 또는 영향 범위 (예: I-009, server.py)
```

- **타입**: `feat`(기능) · `fix`(버그) · `docs`(문서) · `refactor`(동작 변화 없는 구조 개선) · `chore`(설정·잡일)
- **금지**: "수정함" · "업데이트" · "ㅇㅇ" 같은 모호한 메시지. 무엇을·왜가 없으면 안 된다.

**예시**
```
feat: 5축 채점 점수 클램핑 추가 (I-009)

- 모델이 가끔 배점(100) 초과 점수를 반환해 리포트가 깨지는 문제
- evaluate()에서 각 축 점수를 0~배점 범위로 클램핑 처리
- 영향: server.py /interview/evaluate, 응답 스키마 변경 없음
```

## 4. 커밋 전 체크리스트 (60초)

- [ ] 내가 바꾼 파일이 **내 담당**인가? (소유권 맵 5번)
- [ ] **지시받은 것만** 바꿨나? (그 외 변경 0개)
- [ ] `.env`·키·가중치·대용량 파일이 안 섞였나? (`git status`로 확인)
- [ ] `main`이 아니라 **내 브랜치**인가?
- [ ] 커밋 메시지에 **무엇을·왜·영향 범위**가 들어갔나? (3번 형식)

하나라도 NO면 커밋하지 말고 정리부터.

## 5. 파일·모듈 소유권 맵 (자기 영역만 수정)

기능 리스트의 기능 ID(접두어) 기준. **본인 ID 범위의 기능만** 건드린다.

| 담당 | 영역 (기능 ID) | 주 모듈 |
|---|---|---|
| **김진희** (팀장) | `AU-*`, `BE-001~004/006`, `K-001/003/004/005`, `R-003`, `MY-002/003/004`, `I-018`, `B-002`, `A-*` | 인증·기본베이스·공고캘린더·결제·구독·관리자 전체·알림·신고 |
| **정성채** | `BE-005`, `I-001~003`, `I-005~009`, `I-015`, `MY-001/005`, `B-001`, `A-009` | AI 면접엔진·꼬리질문·STAR·5축채점·리포트·추천알고리즘·챗봇응답 |
| **박서진** | `I-004`, `I-010~014`, `I-016/017` | 카메라/음성 테스트·STT·WPM·영상녹화·시선표정·결과PDF·기업추천 |
| **김재원** | `E-001~005`, `K-002/006/007` | 교육 학습 콘텐츠·진행률·교육 캘린더·체크리스트 |
| **김훈기** | `E-006~011`, `MY-007` | AI 퀴즈·정답률·취약개념·경력대비분석·역량 레이더차트 |
| **김윤수** | `R-001/002/004`, `AU-008/009`, `MY-006` | 이력서 작성·히스토리·헤더(찜/히스토리)·My 이력서 |
| **이은총** | `C-001` (+ 질문 아카이브) | 자유게시판·질문 은행 |

> 정확한 **폴더 경로**는 각 레포 구조 확정 후 채운다. 경계가 애매한 파일(공통 컴포넌트·공용 util·DB 스키마)은 **공통 영역**으로 보고 반드시 리뷰 후 변경.

## 6. 새 클래스를 만들기 전 (pre 규칙)

1. **같은 개념이 이미 있는지 먼저 확인** → 있으면 그 이름을 재사용한다. (제각각·중복 클래스 방지)
2. 새 이름이면 **접미사 규칙만** 따른다 — `~Controller` · `~Service` · `~Repository` · `~Dto` · `~Request` · `~Response` (백엔드) / React 컴포넌트는 PascalCase, 훅은 `use~`.
3. **기존 클래스·파일 이름은 담당자 승인 없이 변경 금지.**

**이름 추천 — 채널별 동작**

- **Claude Code**: 레포를 직접 검색해 **실제로 안 겹치는** 이름을 확인하고 추천한다. (충돌 없는 이름 보장)
- **웹(claude.ai)**: 레포를 볼 수 없으므로, **먼저 사용자에게 "기존 클래스 목록(또는 해당 패키지 파일 목록)을 붙여달라"고 요청**한다 → 받은 목록 기준으로 안 겹치게 추천. 목록을 못 받으면 위 규칙 기반으로만 추천하고 *"실제 중복은 확인하지 못했으니 커밋 전에 검색 필요"* 라고 한 줄 고지한다.

## 7. 레포별 적용 (작업 전 판별)

이 저장소가 어떤 레포인지 **파일 존재로 판별**하고, 해당 규칙을 따른다.

| 판별 기준 (루트에 존재) | 레포 | 핵심 추가 규칙 | 넣을 `.gitignore` |
|---|---|---|---|
| `requirements_lock.txt` · `server.py` | **AI** | 아래 고정 스택 절대 변경 금지 / `server.py`·RAG·QLoRA 수정은 **정성채 확인** | `gitignore_AI` |
| `build.gradle` | **백엔드(Spring)** | `application-secret.yml`·`application-local.yml` 커밋 금지 / Gradle `build/` 무시 | `gitignore_backend` |
| `package.json` | **프론트(React)** | `.env.local` 커밋 금지 / `node_modules` 무시 | `gitignore_frontend` |

각 레포 루트에는 **그 레포에 맞는 `.gitignore` 한 개**를 넣는다(파일명을 `.gitignore`로 바꿔 넣는다). `.gitignore`는 git이 글자 그대로 읽으므로 자동 분기되지 않는다 — **누군가 골라 넣어야 한다.**

**최초 셋업 — `.gitignore` 자동 선택 (Claude Code 전용)**
`gitignore_AI` · `gitignore_backend` · `gitignore_frontend` 3개를 모두 받아둔 뒤, Claude Code에 아래처럼 지시하면 판별·복사·이름변경까지 자동으로 된다:

```
이 레포의 스택을 루트 파일로 판별해(package.json→프론트, build.gradle→백엔드,
server.py/requirements_lock.txt→AI), gitignore_* 3개 중 맞는 하나만 골라
레포 루트에 .gitignore 로 복사해줘. 나머지 2개와 다른 파일은 건드리지 마.
```

> 웹(claude.ai)은 레포에 파일을 직접 넣을 수 없다 → 판별 기준만 위 표로 확인하고 **사람이 직접** 맞는 파일을 `.gitignore`로 넣는다.

**AI 레포 — 고정 스택 (절대 변경 금지)**
호환성이 검증된 고정값. 업그레이드·다운그레이드·`revision` 변경 금지.
- `torch 2.4.1+cu124` · `transformers 4.48.3` · `tokenizers 0.21.4` · `accelerate 1.2.1` · `bitsandbytes 0.45.5`
- 모델: `LGAI-EXAONE/EXAONE-Deep-7.8B`, **`revision='e3f42b18f6b1'` 고정**
- 변경이 꼭 필요하면 → 별도 브랜치 + 정성채 리뷰. 바꾼 뒤 **2회 안에 문제 생기면 즉시 롤백.**

**백엔드 레포 (Spring) — 고정 스택**
`DevReady_backend_admin` · `DevReady_backend_user` 공통. 아래 버전·구성으로 통일한다. (버전 변경, 특히 Spring Boot·Java는 별도 브랜치 + 팀 합의 후에만)
- **Java 21** (Oracle OpenJDK 21) · **Spring Boot 3.5.15** · **Gradle (Groovy)** · **MySQL 8**
- 빌드·패키징: Gradle Groovy DSL · 패키징 **Jar** · 설정 파일은 **YAML**(`application.yml`)
- 그룹·패키지: 그룹 `com.devready` / 패키지 `com.devready.<repo>` (예: `com.devready.backend_admin`)
- 주요 의존성: Spring Web · Spring Security · MyBatis Framework · JDBC API · MySQL Driver · Lombok · Spring Boot DevTools · Spring Configuration Processor
- 비밀값은 `application-secret.yml`·`application-local.yml`에 두고 **커밋 금지** (5번·7번 표 준수)

> 프론트(React)의 레포별 세부 규칙은 담당자가 이 표에 이어서 보강한다.

## 8. 웹(claude.ai) 사용자용 — Claude에게 줄 작업 지시 템플릿

> **Claude Code(터미널/IDE)를 쓰면 이 `CLAUDE.md`가 자동 로드되므로 이 템플릿은 불필요하다.**
> 아래는 자동 로드가 안 되는 **웹(claude.ai)** 사용자가 복붙해서 쓰는 용도다. (작업마다 달라지는 범위 지정이 핵심)

```
[작업] <한 줄 목표>
[수정 허용 파일] <정확한 파일 경로 1~3개만>
[금지] 위 파일 외 어떤 파일도 수정·삭제·이동 금지.
[끝나면] 변경한 파일 목록과 diff 요약만 보여줘. git push 하지 마 — 내가 직접 확인 후 푸시한다.
[막히면] 진행하지 말고, 무엇이 막혔는지 질문해.
```

---

## 9. 막혔을 때

추측해서 진행하지 않는다. **멈추고**, 무엇이 막혔는지 / 어떤 결정이 필요한지를 담당자(또는 팀장 김진희)에게 묻는다.
되돌릴 수 없는 작업(삭제·강제푸시·스택 변경) 직전에는 **반드시 한 번 더 사람에게 확인.**
