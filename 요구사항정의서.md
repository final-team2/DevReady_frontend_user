### AU-003 계정 통합 정책

SNS 로그인(카카오·네이버) 및 일반 가입 계정 간 통합 규칙. SNS↔SNS(카카오↔네이버) 통합도 동일하게 적용한다.

| 상황 | 처리 |
|---|---|
| 이메일이 **같음** | 무조건 자동 통합 (별도 인증 불필요) |
| 이메일이 **다름** | ① 전화번호 매칭 → ② 동일 시 통합 진행 여부 확인 → ③ **기존 이메일 인증 성공 후** 통합 |

- 통합 추적을 위해 `member_sns`에 `sns_email`(제공자가 전달한 이메일), `verified_at`(전화번호 매칭 통합 시 인증 시각)을 보존한다.
- 같은 provider로 중복 연동 방지: `UNIQUE(member_id, provider)`, `UNIQUE(provider, provider_uid)`.

- 댓글(depth 0) → 대댓글(depth 1)까지만 허용. **대댓글에 다시 대댓글(대대댓글) 작성 불가.**

### 만족도 조사 정책

- **트리거**: 면접 1회차 완료 직후 1번, 이후 누적 완료 세션 기준 5회마다(6·11·16회…) 노출. 트리거 판정은 백엔드 로직(완료 세션 수 카운트)으로 처리한다.
- **평가 항목 5종**: 질문 적합성(QUESTION_FIT) · 피드백 정확성(FEEDBACK_ACCURACY) · 답변 분석 정확성(ANALYSIS_ACCURACY) · 시스템 편의성(USABILITY) · 전반적 만족도(OVERALL). 각 항목 0~5점.
- 같은 트리거 세션 중복 조사 방지: `UNIQUE(member_id, session_id)`.
- 수집된 만족도는 관리자 영역 A-010(만족도 관리)에서 분석한다.

### 광고·노출 영역 구분

### 공고 작성 방식

- 공고의 자격요건(`requirements`)·우대사항(`preferred_qualifications`)은 **CKEditor 5**로 작성하며, 생성된 HTML을 TEXT 컬럼에 그대로 저장한다.
- 사용자 화면에 출력할 때는 **반드시 sanitize(예: DOMPurify) 후 렌더링**해 XSS를 방지한다. (관리자만 작성하더라도 출력 단계에서 정제)

### 회원 상태·제재 정책

- 회원 상태(`member.status`): `ACTIVE` · `DORMANT` · `SUSPENDED` · `BANNED` · `WITHDRAWN`. 관리자가 회원 정보 페이지에서 직접 변경할 수 있다.
- 제재 유형(`sanction_type`): `WARNING`(경고) · `SUSPEND`(정지) · `BAN`(영구 정지). SUSPEND/BAN은 status를 동기화하고, WARNING은 status를 바꾸지 않고 누적 카운트만 올린다.
- **경고 누적 자동 제재** (`member.warning_count` 기준):

  | 누적 경고 | 자동 조치 |
  |---|---|
  | 1경고 | 해당 콘텐츠 숨김 |
  | 3경고 | 게시 기능 7일 정지 |
  | 4경고 | 게시 기능 30일 정지 |
  | 5경고 | 계정 정지 |

- 제재 근거(`report_id` / `filter_log_id` / `admin_id`) 중 최소 하나는 채워져야 한다. BAN은 종료일(`sanction_end_date`)을 NULL로 둔다.

### 신고·필터링 정책

**신고 사유 코드** (`reason_code`)

| 코드 | 의미 |
|---|---|
| ABUSE | 욕설/비하 |
| SPAM | 광고/도배 |
| OBSCENE | 음란물 |
| PERSONAL_INFO | 개인정보 노출 |
| IRRELEVANT | 주제와 무관 |
| ETC | 기타 (reason_detail에 상세 입력) |

**신고 접수 후 고지**

| 시점 | 신고자 | 피신고자 |
|---|---|---|
| 접수 직후 | 접수 완료 알림 | 고지 없음 |
| 처리 완료(승인) | 처리 결과 알림 | 경고/제재 알림 |
| 처리 완료(기각) | 기각 알림 | 고지 없음 |

**욕설 자동 필터링 (2단계)**

1. **키워드 사전**(`banned_word`, 관리자 관리) 매칭 → 저장 차단 또는 자동 숨김
2. **AI 필터**(EXAONE) → 유해 판단 → 자동 숨김

탐지 결과는 `content_filter_log`에 기록한다(필터 종류 KEYWORD/AI, 조치 BLOCKED/HIDDEN).

### 알림 처리 방식

- 알림 발송은 **Kafka**(프로듀서/컨슈머)로 처리한다. `notification` 테이블은 최종 알림의 **영속 저장소** 역할을 유지한다.
- 알림 유형(`notification_type`)은 DB CHECK 없이 코드(Enum)로 관리한다 (유형이 계속 늘어남).
- 알림 클릭 시 이동을 위해 `target_type`/`target_id`를 둔다 (예: 댓글→게시글, 결제→결제내역). SYSTEM 알림은 NULL.
